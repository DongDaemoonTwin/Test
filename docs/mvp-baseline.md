# MVP 기준선

## 제품 한 문장

항공권·호텔을 예약하기 전에, 내 예산·일정·동행자·취향 기준으로 갈 만한 여행지를 좁혀주고 후보끼리 비교해 최종 선택을 도와주는 서비스.

## MVP의 핵심 역할

이 서비스는 예약 플랫폼이 아니라 여행지 선택 의사결정 도구다.

사용자는 항공권이나 호텔을 예약하기 전에 다음 질문에서 막힌다.

- 내 예산과 일정으로 어디까지 갈 수 있는가?
- 후보 여행지 중 무엇이 더 현실적인가?
- 왜 이 여행지가 내 조건에 맞는가?
- 어떤 여행지는 왜 애매하거나 어려운가?

MVP는 이 질문에 빠르게 답하는 것을 목표로 한다.

## 첫 화면 입력값

| 항목 | 타입 | 예시 | 필수 여부 |
|---|---|---|---|
| 출발지 | string | 인천, 김포, 도쿄, 오사카 | 필수 |
| 여행 월 | number | 7, 8, 12 | 필수 |
| 여행 기간 | number | 3박 4일 | 필수 |
| 예산 금액 | number | 800000 | 필수 |
| 예산 통화 | enum | KRW, JPY, USD | 필수 |
| 예산 단위 | enum | per_person, total | 필수 |
| 동행자 | enum | solo, couple, friends, family, parents | 필수 |
| 비행시간 허용 | number/null | 3, 6, null | 선택 |
| 여행 스타일 | string[] | 가성비, 맛집, 쇼핑, 자연, 문화, 액티비티, 휴양, 사진/SNS, 로컬 경험 | 필수 |
| 꼭 원하는 조건 | string[] | 바다, 쇼핑, 짧은 비행, 치안 | 선택 |
| 피하고 싶은 조건 | string[] | 우기, 장거리 비행, 비싼 물가 | 선택 |

## 결과 구조

결과는 하나의 정답만 보여주지 않고 3개 버킷으로 나눈다.

### 1. 갈 만한 여행지

사용자 조건에 잘 맞는 도시.

예시:

> 후쿠오카  
> 3박 4일, 1인 80만 원, 음식·쇼핑 중심 여행에 적합. 비행시간도 짧고 일정 부담이 적음.

### 2. 애매한 여행지

매력은 있지만 예산, 날씨, 이동시간, 일정 중 한두 가지 리스크가 있는 도시.

예시:

> 다낭  
> 휴양에는 좋지만, 해당 월에 날씨 리스크가 있고 항공권 가격에 따라 예산 초과 가능성 있음.

### 3. 어려운 여행지

여행지 자체는 좋지만 현재 조건과 크게 맞지 않는 도시.

예시:

> 파리  
> 여행지 자체는 좋지만 3박 4일과 1인 80만 원 기준에서는 비행시간·비용 모두 부담이 큼.

## 첫 MVP 페이지

| 페이지 | 역할 | 우선순위 |
|---|---|---|
| `/` | 조건 입력 | P0 |
| `/results` | 갈 만함 / 애매함 / 어려움 결과 | P0 |
| `/compare` | 선택한 도시 2~5개 비교 | P0 |
| `/city/[cityId]` | 도시 상세 정보 | P0.5 |
| `/saved` | 저장한 여행지 | P1 |
| `/feedback` | 사용자 피드백 | P1 |
| `/recommendation` | 독립 최종 추천 페이지 | P1 또는 제거 |

처음에는 `/recommendation`을 별도 페이지로 만들지 않고 `/results` 또는 `/compare` 하단의 최종 추천 메모로 처리한다.

## 첫 개발 순서

1. Google Sheets에서 도시 데이터를 가져와 `Destination` 객체로 정규화한다.
2. 조건 입력값을 `UserTripCondition` 객체로 만든다.
3. 각 도시마다 점수를 계산한다.
4. 점수에 따라 `good`, `borderline`, `difficult`로 분류한다.
5. 결과 카드에 추천 이유와 주의점을 보여준다.

## 화면 표시 원칙

내부적으로는 점수를 계산하되, 사용자에게는 숫자보다 별점, 이유, 주의점을 중심으로 보여준다.

예시:

> 추천도 ★★★★☆  
> 이유: 짧은 일정에 적합하고, 음식·쇼핑 선호와 잘 맞으며, 예산 범위 안에 들어올 가능성이 높습니다.  
> 주의: 성수기에는 항공권 가격이 올라갈 수 있습니다.
